**Goal:** Create the hackathon-ready live integration layer for durable Supabase memory and read-only SAP OData proof while preserving Recoup's deterministic, cited, human-approved contract.

## Required Secret Placement

Use `.env.local` or shell environment only. Do not place secrets in `.env.example`, docs, tests, screenshots, or chat.

For runtime Supabase memory:

```dotenv
SUPABASE_URL=<project-url>
SUPABASE_SERVICE_ROLE_KEY=<service-role-key>
RECOUP_SUPABASE_MEMORY_TABLE=recoup_memory_records
```

When `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are present, Recoup treats Supabase as the primary shared memory backend. `RECOUP_MEMORY_BACKEND=supabase` is no longer required for that selection. SQLite remains the explicit offline fallback when only `RECOUP_MEMORY_DB_PATH` is configured.

For programmatic table creation by Codex, add exactly one DDL channel:

```dotenv
SUPABASE_DB_URL=<postgres-connection-string>
```

Accepted alternatives if no DB URL is shared:

- Run the generated SQL in the Supabase Dashboard SQL Editor.
- Link Supabase CLI locally and run a migration.
- Provide an already-approved privileged SQL RPC name. Do not create such an RPC from REST unless it already exists and is approved.

## Supabase Schema

The approved hackathon schema is intentionally small:

- `recoup_memory_records`: durable operational memory records used by cockpit `/run` and `/memory`.
- `recoup_app_principals`: server-side auth/principal scaffold for future MCP/cockpit service principals.

The SQL is generated by:

```powershell
npm.cmd exec -- tsx -e "import { buildSupabaseMemorySchemaSql } from './src/memory/supabaseStore.ts'; console.log(buildSupabaseMemorySchemaSql('recoup_memory_records'));"
```

The generated SQL must include:

- Memory columns: `sequence`, `id`, `category`, `trust_level`, `scope`, `payload_json`, `record_ids_json`, `created_at`.
- Checks mirroring `src/memory/schema.ts`: allowed categories, allowed trust levels, object JSON payload, non-empty array record IDs.
- Indexes: unique `id`, `(scope, sequence)`, GIN `record_ids_json`, GIN principal `capabilities`.
- RLS: enabled and forced on both tables.
- Grants: no `anon`/`authenticated` access; `service_role` server access only.
